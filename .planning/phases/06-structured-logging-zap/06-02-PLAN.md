<log_mapping>
From CONTEXT.md decisions D-69 through D-72:
- Debug: detailed ffmpeg args, frame processing info, internal state
- Info: user-facing status messages (recording start/stop, progress summary)
- Warn: recoverable issues (retries, partial failures)
- Error: fatal errors (connection failures, invalid config, ffmpeg crashes)
- Progress display: stays on stdout (D-76) - do NOT change these

Current fmt.Println patterns to replace:
1. "[INFO] Starting rtsp-recorder..." → Logger.Info()
2. "[INFO] Using URL from command line..." → Logger.Info()
3. "[INFO] Checking FFmpeg installation..." → Logger.Info()
4. "[INFO] FFmpeg found..." → Logger.Info()
5. "[INFO] Recording configuration..." → Logger.Info() with structured fields
6. "[INFO] Starting recording..." → Logger.Info()
7. "[INFO] Retry X/Y after..." → Logger.Warn() (per D-71 - retries are warnings)
8. "[INFO] Output file: ..." → Logger.Info()
9. "[INFO] Stopping recording: ..." → Logger.Info()
</log_mapping>

**User Decisions:**
- D-75: Replace all fmt.Println/fmt.Printf calls with appropriate zap level
- D-76: Progress display stays on stdout (not logged) - preserve fmt.Printf with \r
- D-77: Error messages to stderr remain visible even at "error" log level
</context>
